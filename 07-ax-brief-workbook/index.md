# AX 과제 브리프 워크북
{: .fs-8 }

**내가 가져갈 일**을 1장으로 정리해 5/27 발표의 출발점으로 삼습니다.
{: .fs-5 .fw-300 }

---

## 이 워크북이 다루는 것

5/20에 들은 강의 + 5/20에 본 3개 하네스 실습을 **본인이 매일 하는 일** 위에 얹는 과정입니다.

- **거창한 부서 전략을 세우지 않습니다.**
- **내 책상 위 일 한 가지**를 골라, 4주 동안 실험할 수 있게 1장으로 정리합니다.
- 이 1장이 곧 5/27 발표 7구조의 기반이 됩니다.

---

## 워크북 8쪽 구조

### 📄 1쪽 — 후보 좁히기
> 본인이 매일/매주 하는 일 중 **자동화·도움받고 싶은 일** 3개를 적고, 5/27까지 4주 검증 가능한 1개로 좁힙니다.

| 후보 | 빈도 | 시간 소요 | 자동화 효과 (1-5) | 데이터 민감도 (1-5) |
|---|---|---|---|---|
| 후보 1 | | | | |
| 후보 2 | | | | |
| 후보 3 | | | | |

좁히기 기준:
- 효과 - 민감도 = **순점수가 가장 높은** 1개 선택

### 📄 2쪽 — AS-IS 흐름
> 현재 어떻게 일하는지 다이어그램으로 그립니다.

```
[시작: 누가/무엇으로]
   ↓
[1단계: 무엇을 한다]
   ↓
[2단계: 어떤 결정]
   ↓
[3단계: 결과물]
```

각 단계마다:
- 누가
- 평균 소요 시간
- 자주 발생하는 실수·시간 낭비

### 📄 3쪽 — TO-BE AI Native 흐름
> 본인 하네스가 적용된 모습.

```
[입력: 본인이 자연어 명령]
   ↓
[하네스: 5명의 AI 직원이 일함]
   ↓
[reviewer: 교차 검증]
   ↓
[결과물: 본인이 검토만]
```

### 📄 4쪽 — 5+3 구성 설계
> 본인 하네스의 agents 5명과 skills 3개를 적습니다.

**Agents (5명)**
1. 전문가 1: _________________ (역할: 데이터·수집)
2. 전문가 2: _________________ (역할: 분석)
3. 전문가 3: _________________ (역할: 위험·이상치)
4. 전문가 4: _________________ (역할: 보고서 작성)
5. reviewer: _________________ (역할: 검증)

**Skills (3개)**
1. 오케스트레이터: 위 5명을 어떤 순서로 일하게 할지
2. 확장 스킬 1: 도메인 방법론
3. 확장 스킬 2: 검증 방법론

### 📄 5쪽 — 비민감 데이터 준비
> 본인 일의 데이터를 어떻게 가공해 AI에 줄지.

| 원본 데이터 | 민감도 | 가공 방법 | 가공 후 예시 |
|---|---|---|---|
| 고객명 | 높음 | 익명화 (C001) | 고객 C001 |
| 매출액 | 중간 | 그대로 또는 비율화 | 100M → 1.0 |
| 지점명 | 낮음 | 그대로 | 강남점 |

### 📄 6쪽 — 테스트 질문 3-5개
> 본인 하네스가 잘 동작하는지 확인할 질문들.

1. 기본 테스트: 평범한 입력에 대한 정상 출력
2. 엣지 케이스: 데이터가 부족한 경우
3. 오류 처리: 잘못된 입력에 대한 대응
4. (선택) 대량 처리: 많은 데이터가 들어왔을 때
5. (선택) 일관성: 같은 입력을 여러 번 돌렸을 때

각 질문마다:
- 입력 예시
- 기대 출력
- 실제 결과 (1주일 동안 채워나감)

### 📄 7쪽 — 4주 검증 계획
> 5/27 발표 후 4주 동안 본인이 책상에서 돌릴 계획.

| 주차 | 무엇을 할 것인가 | 검증 기준 |
|---|---|---|
| 1주차 | 매일 1회 실행, 결과 점검 | 결과물 품질 70% 이상 |
| 2주차 | 동료 1명에게 시연·피드백 | 동료가 "쓰고 싶다" 한 명 |
| 3주차 | 개선 반영 + 본인 외 1명 함께 사용 | 함께 사용 가능 |
| 4주차 | 결과 정리 → 본부 보고 후보 | 본부에 보고 가능한 결과 |

### 📄 8쪽 — 90일 확장 조건
> 검증이 잘 되면 90일 후 어떻게 확장할지 한 줄.

- "4주 검증 결과 ___가 달성되면, 90일 안에 ___까지 확장"
